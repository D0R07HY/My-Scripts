# Student Cyber Security Roadmap

## Roadmap นี้เหมาะกับใคร

Roadmap นี้ออกแบบมาสำหรับคุณโดยเฉพาะในภาพประมาณนี้:

- เป็นนักศึกษาปี 2
- เพิ่งเริ่มเขียน Java และ C ระดับพื้นฐาน
- มีพื้นฐาน Network เล็กน้อย
- สนใจสาย Cyber Security
- อยากมีโปรเจคที่ทำได้จริงและเอาไว้ลง GitHub ได้

นี่ไม่ใช่ roadmap สำหรับคนที่ต้องการไปทำ reverse engineering ระดับลึกทันที

แต่มันคือ roadmap สำหรับคนที่อยาก:

- เริ่มต้นให้ถูกทาง
- ค่อย ๆ สร้างพื้นฐาน
- มีผลงานที่อธิบายได้
- ทำ portfolio ที่ดูจริงจังขึ้นทีละขั้น

## ภาพรวมแบบสั้นมาก

ถ้าสรุปให้สั้นที่สุด เส้นทางของคุณควรเป็นแบบนี้:

1. ตั้งแล็บให้ปลอดภัย
2. เรียนรู้การจดบันทึกแบบนักวิเคราะห์
3. เขียนสคริปต์ช่วยวิเคราะห์ง่าย ๆ
4. เริ่มเขียน YARA และ Sigma จากสิ่งที่สังเกตได้
5. สร้าง simulator ที่ปลอดภัย
6. รวมทั้งหมดเป็น mini portfolio project

พูดง่าย ๆ คือ:

เริ่มจาก "ดูและบันทึก" ก่อน
ค่อยไป "วิเคราะห์"
แล้วค่อยไป "ตรวจจับ"
ก่อนจะรวมเป็น "โปรเจคที่เล่าได้"

## เป้าหมายปลายทาง

หลังจากทำ roadmap นี้ครบ คุณควรมีของประมาณนี้ใน GitHub:

- เอกสารตั้ง lab ที่อ่านแล้วเข้าใจว่าคุณทำงานอย่างปลอดภัย
- report template และ IOC template ที่ใช้ซ้ำได้
- Python scripts ง่าย ๆ สำหรับช่วยงาน analysis
- YARA rules เบื้องต้น
- Sigma rules เบื้องต้น
- simulator ที่ปลอดภัย 1 ตัว
- report จริงอย่างน้อย 1-2 ชิ้น

ถ้ามีสิ่งเหล่านี้ครบ คนที่เข้ามาดูรีโปจะเห็นทันทีว่า:

- คุณไม่ได้แค่ "สนใจ"
- แต่คุณกำลังฝึกแบบเป็นระบบ
- และคุณเข้าใจเรื่องความปลอดภัยกับขอบเขตการใช้งาน

## สิ่งสำคัญที่ต้องเข้าใจก่อนเริ่ม

ตอนนี้คุณยังไม่ต้องรีบไปทาง:

- เขียน malware
- ทำ persistence
- ทำ evasion
- ทำ process injection
- reverse assembly ลึก ๆ

เหตุผลไม่ใช่เพราะคุณไม่มีทางทำได้

แต่เพราะถ้าข้ามขั้นเร็วเกินไป คุณจะเจอปัญหาแบบนี้:

- อ่านไม่เข้าใจว่ากำลังเกิดอะไร
- ทำตามคนอื่นได้ แต่เล่าเองไม่ได้
- โค้ดหรือโปรเจคดูน่ากลัว แต่ความเข้าใจข้างในยังไม่แน่น

เป้าหมาย 3-6 เดือนแรกของคุณควรเป็น:

"ทำของที่ไม่ซับซ้อนเกินไป แต่เข้าใจจริง อธิบายได้จริง และต่อยอดได้จริง"

## วิธีใช้ roadmap นี้

ในแต่ละ phase ผมจะบอก 5 อย่าง:

- คุณกำลังฝึกอะไร
- ทำไมมันสำคัญ
- ต้องทำอะไรบ้าง
- ผลลัพธ์ที่ควรมี
- คุณจะโตขึ้นด้านไหน

ให้คุณคิดว่าแต่ละ phase คือ "เลเวล" ของตัวเอง

ไม่ต้องรีบข้ามเลเวล

## Phase 1: ตั้งฐานให้ปลอดภัยและเป็นระบบ

### ช่วงเวลา

สัปดาห์ 1-2

### เป้าหมายของช่วงนี้

ทำให้ตัวเองมี "ห้องทดลอง" และ "วิธีทำงาน" ที่ปลอดภัยก่อน

ถ้าเปรียบเทียบง่าย ๆ:

ก่อนจะเป็นนักวิเคราะห์ คุณต้องมีโต๊ะทำงานที่เป็นระเบียบก่อน

### คุณกำลังฝึกอะไร

- mindset เรื่อง lab safety
- การแยกระบบจริงออกจากระบบทดลอง
- การทำเอกสารเทคนิคแบบเบื้องต้น

### ทำไมช่วงนี้สำคัญ

ถ้าคุณเริ่มโดยไม่มีระบบ lab ที่ชัดเจน คุณจะเสี่ยง:

- เอาไฟล์ไปทดลองผิดที่
- ไม่รู้ว่าควร revert ยังไง
- ไม่มี pattern ในการจดสิ่งที่เจอ

นักศึกษาหลายคนรีบไปเล่นของยาก แต่ไม่มี process ที่ดี

สิ่งที่ทำให้ดู "จริงจัง" ไม่ใช่ความโหดของเครื่องมืออย่างเดียว
แต่มันคือความเป็นระบบในการทำงานด้วย

### งานที่ควรทำ

1. เขียนเอกสาร `docs/lab-setup/overview.md`
2. ระบุว่าจะใช้ VM อะไร
3. ระบุว่าจะใช้ network แบบไหน
4. ระบุ snapshot workflow
5. เขียน safety rules ของตัวเอง
6. ทำ report template
7. ทำ IOC template

### งานในรีโปที่คุณมีแล้ว

- `docs/lab-setup/overview.md`
- `docs/reports/report-template.md`
- `docs/iocs/ioc-template.md`

### ผลลัพธ์ที่ควรมีเมื่อจบ phase นี้

- คุณอธิบายได้ว่า lab ของตัวเองทำงานยังไง
- คุณรู้ว่าห้ามทำอะไร
- คุณมี template พร้อมใช้สำหรับงานรอบต่อไป

### ทักษะที่โตขึ้น

- การคิดแบบมีขอบเขต
- การเขียนเอกสารเทคนิค
- การวาง workflow

### เช็กตัวเองก่อนข้าม phase

ถ้ามีคนถามว่า:

"ถ้าจะทดลองอะไรสักอย่าง คุณจะทำบนเครื่องไหน ใช้ network แบบไหน แล้วคืนสภาพยังไง"

คุณควรตอบได้ชัดเจน

## Phase 2: เริ่มเป็นนักวิเคราะห์จากงานเล็ก ๆ

### ช่วงเวลา

สัปดาห์ 3-5

### เป้าหมายของช่วงนี้

ฝึกอ่านไฟล์ในมุมของนักวิเคราะห์ ไม่ใช่ในมุมของผู้ใช้ทั่วไป

### คุณกำลังฝึกอะไร

- hashing
- string extraction
- metadata awareness
- การมองหา artifact ที่มีประโยชน์

### ทำไมช่วงนี้สำคัญ

นี่คือจุดเปลี่ยนจาก:

"ฉันมีไฟล์อยู่ไฟล์หนึ่ง"

ไปสู่:

"ฉันรู้ว่าจะเริ่มสำรวจไฟล์นี้ยังไง"

งานพื้นฐานแบบนี้สำคัญมาก เพราะมันคือรากของ static analysis

### งานที่ควรทำ

1. ใช้ `hash_report.py` กับไฟล์ตัวอย่าง
2. ใช้ `basic_strings.py` กับไฟล์ text หรือ binary ง่าย ๆ
3. จดผลลัพธ์ลง report template
4. ลองสรุปว่าข้อมูลอะไร "น่าจะเป็น IOC"

### งานในรีโปที่คุณมีแล้ว

- `analysis/scripts/hash_report.py`
- `analysis/scripts/basic_strings.py`

### ควรทดลองกับอะไรดี

เริ่มจากไฟล์ที่ปลอดภัยก่อน เช่น:

- `LICENSE`
- โปรแกรม test ของตัวเอง
- binary ง่าย ๆ ที่คุณ compile เอง
- sample จากแหล่งสาธารณะที่ไม่อัปโหลดเข้ามาในรีโป

### ผลลัพธ์ที่ควรมีเมื่อจบ phase นี้

- คุณใช้ hash เพื่อระบุไฟล์ได้
- คุณรู้ว่า strings บอกอะไรเราได้บ้าง
- คุณเริ่มเขียน report ได้จากข้อมูลจริง

### ทักษะที่โตขึ้น

- Python scripting เบื้องต้น
- การอ่าน artifact
- การสรุปเชิงเทคนิค

### เช็กตัวเองก่อนข้าม phase

ถ้ามีคนถามว่า:

"ถ้าได้ไฟล์ที่ไม่รู้จักมา 1 ไฟล์ คุณจะเริ่มดูอะไรก่อน"

คุณควรตอบได้ประมาณนี้:

- ดู hash
- ดูขนาดและชนิดไฟล์
- ดึง strings
- จดสิ่งที่น่าสนใจลง report

## Phase 3: เริ่มคิดแบบคนเขียน Detection

### ช่วงเวลา

สัปดาห์ 6-8

### เป้าหมายของช่วงนี้

เปลี่ยนจากแค่ "เห็นข้อมูล" ไปเป็น "คิดว่าจะตรวจจับยังไง"

### คุณกำลังฝึกอะไร

- การเลือก indicator ที่เหมาะกับ detection
- การเขียน logic แบบง่าย
- การคิดเรื่อง false positive

### ทำไมช่วงนี้สำคัญ

หลายคนวิเคราะห์ได้ แต่ยังไม่สามารถเปลี่ยนสิ่งที่เจอให้เป็นกฎตรวจจับได้

แต่ถ้าคุณทำได้ แม้จะเป็นระดับ beginner มันจะทำให้โปรไฟล์ของคุณแข็งแรงขึ้นมาก

เพราะมันแปลว่าคุณเริ่มมองจากมุม defensive engineering แล้ว

### งานที่ควรทำ

1. เขียน YARA rule แรกจาก strings ง่าย ๆ
2. เขียน Sigma rule แรกจาก process behavior ง่าย ๆ
3. ใส่ note ว่า rule นี้ตั้งใจจับอะไร
4. ใส่ note ว่าอาจมี false positive แบบไหน

### จุดสำคัญที่ต้องเข้าใจ

YARA มักใช้กับ:

- file content
- string patterns
- binary or text artifacts

Sigma มักใช้กับ:

- logs
- process creation
- command lines
- event patterns

### ผลลัพธ์ที่ควรมีเมื่อจบ phase นี้

- YARA 1-3 ไฟล์
- Sigma 1-2 ไฟล์
- คำอธิบาย rule ที่คุณเขียนเองได้

### ทักษะที่โตขึ้น

- detection mindset
- logic design
- defensive thinking

### เช็กตัวเองก่อนข้าม phase

ถ้ามีคนถามว่า:

"ทำไมเลือก string นี้มาใส่ใน YARA"

หรือ

"Sigma rule นี้ต้องใช้ log อะไร"

คุณควรตอบได้ด้วยภาษาของตัวเอง

## Phase 4: สร้าง Simulator ที่ปลอดภัย

### ช่วงเวลา

สัปดาห์ 9-10

### เป้าหมายของช่วงนี้

สร้างพฤติกรรมทดสอบที่ "ไม่เป็นอันตราย" แต่มีประโยชน์ต่อการฝึก detection และ reporting

### คุณกำลังฝึกอะไร

- การออกแบบพฤติกรรมจำลอง
- การควบคุม scope ของโปรแกรม
- การเชื่อม script เข้ากับ detection

### ทำไมช่วงนี้สำคัญ

นี่คือจุดที่โปรเจคของคุณเริ่ม "มีเรื่องให้เล่า"

ไม่ใช่แค่มีไฟล์สคริปต์หลายตัว
แต่เริ่มเห็นภาพว่า:

- script นี้ทำอะไร
- report นี้อธิบายอะไร
- rule นี้จับอะไร

### ตัวอย่างโปรเจคที่เหมาะกับคุณ

`Safe File Activity Simulator`

แนวคิดคือ:

- สร้างไฟล์ทดสอบในโฟลเดอร์แล็บ
- เปลี่ยนชื่อไฟล์
- สร้าง log หรือ console output
- ไม่แตะไฟล์จริงของผู้ใช้
- ไม่มี encryption
- ไม่มี network beacon

### งานที่ควรทำ

1. สร้าง simulator ใน `simulators/`
2. จำกัด path ให้ทำงานได้เฉพาะโฟลเดอร์ lab
3. เขียน README ของ simulator
4. จด artifact ที่เกิดขึ้น
5. เชื่อม artifact เหล่านั้นกับ YARA หรือ Sigma ที่เกี่ยวข้อง

### ผลลัพธ์ที่ควรมีเมื่อจบ phase นี้

- simulator 1 ตัว
- note ว่ามันสร้าง artifact อะไร
- detection ที่ใช้คู่กันได้อย่างน้อย 1 ตัว

### ทักษะที่โตขึ้น

- การออกแบบ mini project
- การควบคุมความเสี่ยง
- การคิดแบบ end-to-end

## Phase 5: รวมทุกอย่างให้เป็น Portfolio Piece

### ช่วงเวลา

สัปดาห์ 11-12

### เป้าหมายของช่วงนี้

ทำให้ทุกอย่างที่คุณสร้างมาเชื่อมกันเป็นโปรเจคที่คนอื่นอ่านแล้วเข้าใจ

### คุณกำลังฝึกอะไร

- การเล่าเรื่องทางเทคนิค
- การจัดระเบียบ GitHub
- การสรุปสิ่งที่เรียนรู้

### งานที่ควรทำ

1. เขียน report จริง 1 ชิ้น
2. อธิบายว่า simulator ทำอะไร
3. อธิบายว่า IOC อะไรเกิดขึ้น
4. อธิบายว่า detection ตัวไหนใช้ได้
5. สรุป limitation และสิ่งที่จะพัฒนาต่อ

### ผลลัพธ์ที่ควรมีเมื่อจบ phase นี้

- report 1-2 ชิ้น
- IOC notes 1-2 ชิ้น
- scripts 2-3 ตัว
- detections 2-4 ตัว
- repo ที่ดูมีทิศทางชัดเจน

### ทักษะที่โตขึ้น

- portfolio building
- communication
- self-directed learning

## ถ้ารู้สึกว่ายาก ให้เริ่มแค่ชุดนี้ก่อน

ถ้าคุณรู้สึกว่า roadmap ทั้งหมดใหญ่เกินไป ให้เริ่มจาก mission สั้น ๆ ชุดนี้:

### Mini Mission A

- อ่าน `docs/lab-setup/overview.md`
- เติมข้อมูล lab ของตัวเองให้ครบ

### Mini Mission B

- รัน `hash_report.py` กับไฟล์ 3 ไฟล์
- เปรียบเทียบผลลัพธ์

### Mini Mission C

- รัน `basic_strings.py` กับไฟล์ 2 ไฟล์
- จดว่า strings ไหนน่าสนใจ

### Mini Mission D

- เขียน report แรกจาก public writeup 1 ชิ้น

ถ้าคุณทำ 4 mission นี้ได้ คุณจะเริ่มเห็นภาพทั้งโปรเจคชัดขึ้นมาก

## สิ่งที่คุณควรคาดหวังจากตัวเอง

ในช่วงนี้ อย่าคาดหวังว่าจะต้อง:

- เก่ง reverse engineering มาก
- เขียน detection ขั้นสูงทันที
- รู้ทุกเรื่องใน Cyber Security

สิ่งที่ควรคาดหวังคือ:

- ทำสม่ำเสมอ
- เขียนสิ่งที่ตัวเองเข้าใจจริง

## ถ้าจะใช้ roadmap นี้กับการสมัครฝึกงาน

สิ่งที่ผู้ดูส่วนใหญ่จะมองหา ไม่ใช่แค่ความยากของโปรเจค

แต่คือ:

- คุณจัดระเบียบงานดีไหม
- คุณอธิบายสิ่งที่ทำได้ไหม
- คุณเข้าใจเรื่องความปลอดภัยไหม
- คุณมีแนวโน้มเรียนรู้ต่อเองได้ไหม

roadmap นี้ถูกออกแบบมาเพื่อช่วยให้คุณตอบคำถามพวกนั้นได้ดีขึ้น

## เป้าหมายที่เหมาะที่สุดสำหรับคุณตอนนี้

ถ้าจะสรุปเป็นประโยคเดียว:

"สร้างรีโปที่แสดงให้เห็นว่าคุณเป็น beginner ที่จริงจัง มีวินัย และกำลังโตไปในทางที่ถูก"

นั่นดีกว่าการพยายามทำโปรเจคที่ดูโหด แต่ตัวคุณเองยังอธิบายไม่ได้ว่าข้างในเกิดอะไรขึ้น

## ขั้นต่อไปที่แนะนำทันที

หลังจากอ่าน roadmap นี้แล้ว ผมแนะนำให้คุณทำต่อเป็นลำดับนี้:

1. เติมข้อมูลจริงลงใน `docs/lab-setup/overview.md`
2. รัน `hash_report.py` กับไฟล์ 3 ไฟล์
3. รัน `basic_strings.py` กับไฟล์ 2 ไฟล์
4. เปิด issue แรกจาก `docs/issues/`
5. เริ่มเขียน report แรกจาก public writeup
